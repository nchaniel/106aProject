Vision-Guided Pick and Place
with a UR7e Robot Arm

A closed-loop robotic system that detects food objects on a table with a wrist-mounted Intel RealSense camera, reconstructs their 3D positions in the robot's base frame, and executes MoveIt 2-planned grasps that deposit each object onto a detected plate.

1 · System Overview

The system runs in two phases: an offline inspection phase that builds a multi-view dataset of the scene, and an online execution phase that turns the resulting 6D positions into a sequence of grasps.

Phase 1 — Inspection

  1. Detection node publishes the live plate centroid in base_link.
  2. ArmCircler orbits the arm in two rows of twenty waypoints around the plate, with the camera permanently aimed at the centroid.
  3. At each waypoint the node saves an RGB capture and the corresponding end-effector pose into poses.npy.
  4. Once forty captures are written, /orbit_done latches on a transient-local topic so the commander always picks it up.

Phase 2 — Execution

  1. Commander spawns two subprocesses: segment_batch.py (YOLO + SAM2) and the 6D pose pipeline.
  2. Multi-view triangulation produces results/object_poses.npy with a position_base_link_m for every detected object.
  3. User presses Enter in the commander terminal to activate pick-and-place.
  4. UR7e_CubeGrasp runs the seven-step grasp sequence for each object and returns the arm to its home pose between picks.

2 · Architecture

The pipeline is split across two ROS 2 packages — perception and planning — with MoveIt 2 providing IK and motion planning, and the RealSense ROS 2 driver providing aligned RGB + point-cloud streams.

Design note — The static wrist_3_link → camera_link TF is the bridge between the depth point cloud and the robot's kinematic chain. Without it, YOLO can locate objects in the camera frame but the arm has no way to plan to them.

3 · Perception Pipeline

DetectionNode consumes RealSense RGB, depth, and intrinsics, runs YOLOv8 on every frame, and converts each accepted bounding box into a 3D centroid in base_link. A plate-exclusion filter prevents the arm from treating anything resting on the plate as a pick candidate.

Step-by-step

  1. YOLO inference. YOLODetector.detect() returns class IDs, confidence, bbox, and 2D center.
  2. Depth lift. get_centroid_from_cloud_bbox() reprojects every cloud point into image coords, keeps the ones inside the bbox, and averages them into a PointStamped.
  3. TF lift. transform_point() resolves camera_link → base_link through the static TF + UR kinematic chain.
  4. Plate gate. Any candidate within plate_radius = 0.14 m of the latest plate centroid is rejected so the gripper cannot try to lift objects sitting on the plate.
  5. Publish. The highest-confidence surviving detection is published on /detected_pick_point + /detected_class; the plate centroid is always published on /detected_plate_point.

4 · 6D / Multi-View Pose Estimation

Live single-frame centroids degrade at the edges of the camera frame. To get a stable, frame-independent position for every object we exploit the orbit captures: forty RGB images with synchronized end-effector poses become the input to an offline multi-view pipeline.

Inputs

  • captured_images/ — 40 RGB frames
  • poses.npy — [x, y, z, qx, qy, qz, qw] per frame
  • updated.pt — fine-tuned YOLO weights
  • SAM2 (tiny) for mask refinement

Per-view processing

  • YOLO bbox → SAM2 mask → centroid pixel (u, v).
  • Apparent radius estimate: z = fx · (d/2) / rpx.
  • Back-project (u, v, z) into the camera frame.
  • Compose with T_CAM_FROM_EE (camera +3 cm forward, −13.5 cm below the EE) and the recorded EE pose.

Multi-view fusion

  • Per-object set of camera rays across all views that saw it.
  • triangulate_ransac() rejects outlier views, then triangulate_dlt() solves a weighted linear system.
  • Result: a single 3D centroid in the world (= base_link) frame, plus a confidence equal to the fraction of views that saw the object.

Output

  • results/object_poses.npy
  • Each entry: object_name, instance_idx, position_base_link_m, confidence.
  • Consumed by the commander to plan a pick order and per-object drop spots.
Why position-only? The current pipeline runs in position_only=True mode — it skips DINOv2 features and STL rendering and estimates depth purely from apparent mask size. This is fast (no GPU mesh rendering) and avoids dependence on accurate CAD models for soft, irregular objects like cake or berries.

5 · Planning & Execution

Once the commander hands off, UR7e_CubeGrasp runs a strict job-queue state machine. Every motion is an asynchronous trajectory; gripper toggles are synchronous service calls. The arm always returns home between picks so the next detection sees a known camera vantage point.

  1. Home

    Idle observation pose; gripper open.

  2. Pre-pregrasp

    Move 0.5 m above the rough centroid for a centered camera view.

  3. Refine

    Capture a second detection from the centered view and rebuild the queue with full PICK_OFFSETS.

  4. Hover

    IK at (x+Δx, y+Δy, z + pre_grasp_z_offset).

  5. Descend

    IK at z + grasp_z_offset.

  6. Grip

    /toggle_gripper Trigger service.

  7. Lift

    IK at z + lift_z_offset.

  8. Over plate

    Translate to (plate_x, plate_y, drop_z + 0.20).

  9. Lower & release

    Descend to drop_z + 0.15, toggle gripper.

  10. Home

    _going_home sentinel clears the busy flag once the arm settles.

Key design patterns

busy semaphore

Latches the moment a detection is accepted; blocks every subsequent cube_callback until the arm is home again.

IK chaining

Every compute_ik() uses the previous step's joint solution as its seed, preventing elbow flips between adjacent waypoints.

Two-phase refinement

Edge-of-frame depth distortion is mitigated by computing the grasp from a second, well-centered detection captured at the pre-pregrasp pose.

Async trajectories

_execute_joint_trajectory → _on_goal_sent → _on_exec_done → execute_jobs. The spin loop never blocks on a trajectory.

6 · Current Status & Known Issues

The code is still under active iteration. Perception and orbit inspection are stable; the offline 6D pipeline runs end-to-end; pick execution works on rigid objects; placement on the plate is the next milestone.

Working

  • Live YOLOv8 detection + plate-radius exclusion
  • 40-waypoint orbit with synchronized pose logging
  • SAM2 batch segmentation across all captures
  • Multi-view RANSAC triangulation in base_link
  • Async MoveIt-planned pick sequence with refinement

In progress

  • Wiring position_base_link_m from object_poses.npy into main.py instead of live detection.
  • Computing distinct per-object drop spots around the plate centroid (not all stacking on one point).
  • Sorting picks by distance / object-type priority in the commander.

Known issues

  • Drop-waypoint IK failures are not currently checked before enqueueing — a failed IK passes None into plan_to_joints.
  • Gripper state is assumed open at startup; out-of-sync state silently flips every subsequent toggle.
  • If the object moves or is removed after the arm reaches pre-pregrasp, the refined detection waits indefinitely.
  • Placement on the plate is not yet tuned per object — current drop height is a single constant.
  • colcon build copies (not symlinks) Python files, so source-tree edits must be mirrored to install/ until rebuilt.

8 · Future Work

The biggest gaps are between the offline 6D pipeline and the live executor. Closing them is mostly engineering plumbing rather than new algorithms.

Pick ordering

Sort detected objects by distance-from-home or by class priority before publishing the task list, so each successive grasp leaves the camera's path to the next one clear.

Distinct drop spots

Compute a radial layout around the plate centroid so multiple objects land at different points instead of stacking. Radius ≈ 4–6 cm; angle indexed by pick order.

Robust grasp offsets

Replace the per-class PICK_OFFSETS table with a small calibration routine that measures actual gripper-finger geometry against each object's bounding height.

Failure recovery

Guard against missing post-pregrasp detections, failed drop-waypoint IK, and gripper state drift. Reset to home on timeout instead of waiting indefinitely.

Live refinement / tracking

Replace the single refinement detection with a short visual servo loop during descent — useful when small objects shift while the gripper is closing.

Task-list IPC

Define a custom message that carries (class, source_xyz, drop_xyz) tuples so commander → main.py communicates the planned sequence in a single publish, replacing live detection entirely on the executor side.
